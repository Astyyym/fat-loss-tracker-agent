# Issue #2 — 首次启动档案与减脂计划配置

## 1. 目标

首次打开本地仪表盘时，用户先完成本地档案与计划设置。只有档案有效后才展示正式仪表盘；不得显示维护者预设的周期、热量或蛋白质目标。

## 2. 唯一数据模型

`profile.json` 是唯一私有档案与计划真相来源，包含：

- `schema_version`
- `profile`：身高、当前体重、目标体重，以及可选年龄、生理性别、日常活动水平、每周训练频率
- `plan`：`start_date`、`end_date`、`program_weeks`、热量目标、蛋白质目标、时区

首次设置时，用户在“目标日期”和“计划周数”中二选一：

- 目标日期：保存该日期为 `end_date`，用确认当天与时区计算 `start_date` 和 `program_weeks`。
- 计划周数：保存输入周数，用确认当天与时区计算 `start_date` 和 `end_date`。

修改目标时使用相同规则，更新当前计划配置，但绝不改写历史 JSONL 事件。仪表盘对历史记录按当前计划进行比较时必须标明“当前计划口径”。

## 3. 旧配置迁移

旧 `config.json` 只允许一次性迁移热量、蛋白质、时区等可复用字段。即使发现旧配置，用户仍必须填写身高、当前体重、目标体重及日期/周数；不得跳过首次设置或显示默认目标。

迁移或创建完成后，`profile.json` 是唯一权威来源。

## 4. 后端与网页边界

- `backend/service.py` 保持唯一业务与数据读写入口。
- 通过受限的 localhost 配置 API 读写档案；仪表盘数据 API 仍不提供事件编辑能力。
- 前端不使用 LocalStorage 保存个人或健康数据，也不维护第二套业务真相。
- `profile.json` 损坏时显示明确错误和恢复入口；恢复前备份损坏文件，不静默回退默认值。
- 写入使用校验和原子替换。

## 5. 安全与建议

- 身高、体重、日期、周期必须有合理范围。
- 目标日期不能早于开始日期。
- 根据目标体重与周期计算速度；过激目标必须警告，不能生成极端热量建议。
- 热量与蛋白质建议必须标为“估算建议”，用户可确认并手动调整；不构成医疗建议。
- 真实 `profile.json`、`config.json`、数据、备份、摘要、卡片、日志均必须被 Git 忽略。

## 6. 验收标准

- [ ] 无有效档案时仅显示设置向导，不显示默认仪表盘或固定目标。
- [ ] 提交有效档案后原子写入 `profile.json`，刷新/重启后仍可读取。
- [ ] 用户可选择目标日期或计划周数，后端规范化保存起止日期和周数。
- [ ] 可修改个人目标，历史 JSONL 不变。
- [ ] 页面展示计划起止日期、周期、当前体重、目标体重和当前目标。
- [ ] 非法值、过激目标、损坏档案、迁移行为和配置 API 均有自动化测试。
- [ ] `profile.json`、`config.json` 与健康数据不被 Git 跟踪。
- [ ] 全量测试、`git diff --check`、Windows 启动与配置流程冒烟通过。

## 7. 禁止事项

- 不增加多用户、登录、云同步、数据库、Docker 或 npm。
- 不恢复 Linux、WSL 或 macOS 的公开说明。
- 不直接修改历史 JSONL。
- 不把档案写入 LocalStorage。
- 不硬编码维护者身体数据、周期或绝对路径。
